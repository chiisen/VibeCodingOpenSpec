Spec-driven development for AI coding assistants

OpenSpec 一頁式說明

先對齊需求，再開始寫程式。OpenSpec 讓人類與 AI 協作者在同一份規格上共識，保留審閱軌跡、降低溝通成本。

npm install -g @fission-ai/openspec 檢視 GitHub 專案

免 API key，與 Codex、Cursor、Claude Code 等工具並行使用。使用 `openspec init` 為團隊建立共用規格流程。

為什麼選擇 OpenSpec Why OpenSpec

當需求只存在對話與記憶中，很難確保 AI 助手與人類理解一致。OpenSpec 以輕量規格流程鎖定意圖，帶來可重複、可審閱的成果。

對齊意圖

使用變更資料夾（proposal / tasks / specs）先定義工作範圍，再請 AI 編碼，避免實作偏離原始需求。

可稽核

所有的規格修改都以 delta 紀錄，誰在何時改了什麼一目了然，適合團隊審查與長期維運。

工具友善

支援原生 slash commands（如 Cursor、Windsurf）與 AGENTS.md 指引，可套用在不同 AI coding assistants。

Brownfield-first

區隔「現況」與「提案」，讓維護既有系統或跨規格調整時仍能保持變更清楚易懂。

工作流程 Workflow Overview

四個步驟讓人類與 AI 協作：規劃 → 對齊 → 實作 → 歸檔，每一步都留下明確的規格與任務紀錄。

  1. 1. Draft Change

    建立 `openspec/changes//`，撰寫 `proposal.md`、`tasks.md` 與 delta 規格描述預期行為。

  2. 2. Review & Align

    與 AI 助手或夥伴反覆調整文案與規格，直到所有人對需求與實作步驟都有共識。

  3. 3. Implement Tasks

    依 `tasks.md` 的核取清單實作與測試，必要時在變更資料夾記錄設計決策。

  4. 4. Archive & Update Specs

    完成後執行 `openspec archive `，將核准的 delta 合併回 `openspec/specs/` 作為最新真實來源。

快速開始 Getting Started

環境需求

  • Node.js >= 20.19.0（`node --version`）
  • npm（或其他套件管理工具）
  • 任一支援 AGENTS.md 或 slash commands 的 AI coding assistant

安裝流程

  1. 在專案根目錄執行：npm install -g @fission-ai/openspec@latest
  2. 使用 openspec init 建立結構與 AGENTS 指南。
  3. 閱讀 `openspec/project.md` 並填寫專案背景與技術棧。
  4. 建立第一個變更提案，請 AI 依據規格進行實作。
小提示：使用 `openspec update` 可更新共享指引，確保不同工具的 AI 助手遵循最新規範。

CLI 快速參考 CLI Quick Reference

openspec init

初始化專案結構，生成 AGENTS 指南與預設資料夾。

openspec list

檢視目前所有變更與狀態，掌握提案與實作進度。

openspec validate --strict

驗證變更是否符合格式要求，包含 delta 標頭與 Scenario 區塊。

openspec diff <change-id>

比對變更資料夾內的規格差異，便於審閱與溝通。

openspec apply <change-id>

讓 AI 助手讀取提案並著手實作 TODO 項目。

openspec archive <change-id>

將完成的變更合併回 specs/，並移至 archive/ 作為審計紀錄。

變更資料夾範例 Example Change Folder

openspec/
├─ specs/
│  └─ auth/
│     └─ spec.md              # 現行規格
└─ changes/
   └─ add-two-factor-auth/
      ├─ proposal.md          # 為什麼、預計變更
      ├─ tasks.md             # 實作待辦
      ├─ design.md            # 技術筆記（選填）
      └─ specs/
         └─ auth/
            └─ spec.md        # Delta：新增 MFA 條目

每個變更都保留完整脈絡：proposal.md 說明緣由、tasks.md 描述實作清單、specs/ 撰寫 delta 記錄新增、修改或移除的需求。

Delta 需使用 ## ADDED / MODIFIED / REMOVED / RENAMED Requirements 標題，並確保每個需求至少有一個 #### Scenario: 區塊以 Given/When/Then 方式描述。

完成實作後執行 `openspec archive add-two-factor-auth`，讓核准的規格合併回 openspec/specs/，維持唯一真實來源。
